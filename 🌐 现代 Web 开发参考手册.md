### 我为什么写这个东西？

```
HTML 标签 → CSS 属性 → JavaScript 语法 → 服务器配置 → 数据库连接
```

问题是什么？很多人学完只会“这个标签怎么写”，但还是不知道“一个网站到底是怎么跑起来的”。

现在做 Web 开发，早就不用一个人硬啃几万行代码了，甚至自己完全可以一行代码都不写，只需要审核AI给你的代码。所以现在的模式是：

```
你提出需求 → AI 生成代码 → 你看懂结构 → 你修改维护 → 系统上线
```

1. **搞清楚概念**（这玩意儿是干嘛的）

3. **学会怎么跟 AI 描述需求**（让它帮你干活）
4. **看懂代码在干什么**（方便纠错和改功能）

## 1. Web相关基础知识

### 1.1 Web的本质

**一句话**：Web 就是你用浏览器打开的所有“网站”和“网页应用”——南农的教务系统、知网、ChatGPT 网页版、4399小游戏网页，统统都是。

很多初学者会把 **Internet（互联网）** 和 **Web（万维网）** 搞混。我打个比方：

- **Internet** = 全球高速公路网（光纤、路由器、IP 地址这些基础设施）
- **Web** = 高速公路上跑的车（浏览器、网页、你看到的内容）

---

### 1.2 服务器的含义

"服务器"有三个不同的意思：

**意思①：硬件（一台物理计算机）**

南农机房里的那台主机箱，有CPU、内存、硬盘、网卡。你买的云服务器（阿里云ECS、腾讯云轻量应用服务器）就是这种东西。

**意思②：软件（跑在这台计算机上的程序）**

比如你装了个Nginx（Web服务器）、装了个MySQL（数据库服务器）、跑了你的Flask代码（应用服务器）。这些都是"服务器软件"，运行在硬件服务器上。

**意思③：功能角色（这台计算机提供的一些服务）**

一个硬件服务器上可以同时跑多个角色：

- **Web服务器**（接待请求，处理静态资源）
    
- **应用服务器**（跑业务逻辑，比如你的Flask代码）
    
- **数据库服务器**（存数据）

**关于"服务器"这个词**：下文说的"服务器"默认指运行后端代码的电脑或程序。它不包含前端，前端在浏览器里跑。如果你听到"Web服务器""应用服务器"这些词，它们指的是服务器上跑的不同软件角色。

**本地服务器实际上就是：我的电脑充当服务器。**

---
### 1.3 浏览器——用户接触Web的入口

 **① 浏览器是什么**

浏览器本质上是一个 **"翻译器"** 。它把服务器返回的HTML、CSS、JS代码"翻译"成你看到的图文并茂的页面。没有浏览器，你面对的就是一堆尖括号和字母符号。


常见的浏览器：Chrome（最主流）、Edge、Safari（苹果）。你用哪个都行，但开发时推荐 Chrome 或 Edge（它们自带的开发者工具最顺手）。

 **② 浏览器是怎么工作的**

浏览器的工作原理可以浓缩成四步：**输入 → 请求 → 接收 → 渲染**。

> 你在地址栏敲网址按回车，浏览器就去服务器把页面要的资源拿回来，然后解析HTML搭骨架、应用CSS上样式、执行JS加交互，最终把页面展示给你。


 **③ 浏览器里能干什么（“不止是看网页”）**

按F12打开的开发者工具，是**调试Web应用的核心面板**：

- **Elements（元素）**：看当前页面的HTML结构，可以临时改文字、调样式看效果，关掉页面就恢复原样，不修改源文件。
    
- **Console（控制台）**：直接写JS代码测试逻辑、查看报错信息——开发时80%的调试时间花在这里。
    
- **Network（网络）**：看前端发的每个请求（URL、方法、状态码、返回数据）。联调时接口通不通、返回了什么，全靠它排查。
    
- **Application（应用）**：查看Cookie、localStorage里存了什么东西。


 **④ 浏览器和搜索引擎是两回事**

- **浏览器**：一个软件，用来打开网页。比如Chrome、Edge。
    
- **搜索引擎**：一个网站，用来帮你找到网页。比如百度、Google。
    

你在Chrome（浏览器）里打开百度（搜索引擎）搜索"南农图书馆"，然后点击搜索结果进入南农图书馆官网——这一连串动作同时用到了浏览器和搜索引擎。

### 1.4 Internet运行的两大支撑
 
> TCP/IP协议族

它是分层工作的，从下层到上层大概是：

（1）最底层：网络接口，网线，网卡这些硬件连接

（2）IP层：给每个数据包写好“源地址+目标地址”，也就是IP地址

（3）TCP/UDP层：TCP负责把数据拆成小包，保证对方能够完整收到，丢了就重发。UDP发得快但不保证送达。

（4）最顶层：应用层，我们直接用的功能都在这层。

> DNS域名系统

因为IP地址是一串数字（比如`192.168.1.1`），人记不住，所以设计了**域名**（比如 `www.njau.edu.cn`）来代替数字IP。
DNS就相当于互联网的电话本：你输入好记的域名，DNS服务器帮你查对应的IP地址，电脑才能找到目标服务器。

---
### 1.5 Web的核心技术

（1）URL——统一资源定位器  
就是我们说的网址，相当于互联网上每个文件，图片，网页的精确地址。它告诉浏览器这个资源在哪，用什么方式打开。  

   标准格式：协议://主机名:端口号/路径/文件名?查询参数
  
   例如：
   `https://jwgl.njau.edu.cn/jsxsd/xsxk/index.html?学期=2026春&学生=海诺`
   
（2）HTTP——超文本传输协议  

它是浏览器和服务器之间的对话规则，规定了“请求”和“响应”的格式。工作模式是请求-响应：浏览器向服务器发请求（比如“我要访问这个网页”），服务器收到后返回对应的内容（HTML、图片、JSON数据等）。前面1.4里TCP/IP负责“把数据送到”，HTTP负责“数据里写什么内容”——一个管运输，一个管内容。  

（3）HTML——超文本标记语言  

用来描述网页内容和结构的标记语言。它用各种标签标记文字、图片、链接、表格等内容，告诉浏览器“这段是标题”“那里是图片”。浏览器拿到HTML文件后，解析标签、渲染成可视化页面。**HTML只管内容和结构，不管样式和交互**——样式交给CSS，交互交给JavaScript。  

（4）浏览器

浏览器是用户唯一需要主动操作的那个入口——你输入网址、点击链接、填写表单，所有动作都在浏览器里完成。它负责把 URL、HTTP、HTML 这三样东西整合起来，最终让你看到一个能交互的页面。

## 2. 网页相关知识

### 2.1 网址的本质——服务器地址

**（1）网址就是“服务器的门牌号”**

你平时在浏览器里输入的网址，本质上就是一个**服务器的地址**。它的作用就是告诉浏览器：“你要找的资源，在这台服务器上。”

|      你输入的网址       |    实际含义     |
| :---------------: | :---------: |
|  `www.baidu.com`  |  访问百度的服务器   |
|  `www.cnki.net`   |  访问知网的服务器   |
| `lib.njau.edu.cn` | 访问南农图书馆的服务器 |

> **一句话概括**：网址本身不是内容，它是内容的地址。就像快递单上的收件地址不是包裹本身，但包裹要靠这个地址才能送到你手上。

**（2）本地网址：访问“自己电脑”的地址**

在开发过程中，你还会遇到一类特殊的网址：

|     地址      |         含义          |
| :---------: | :-----------------: |
| `127.0.0.1` |    本机回环地址（固定数字）     |
| `localhost` | 本地主机名（等价于127.0.0.1） |

这两个地址都指向**你自己的电脑**，而不是互联网上的任何服务器。

**它们什么时候用？**

当你在PyCharm里运行Flask应用，终端里显示 `Running on http://127.0.0.1:5000` 时，意思就是：这个服务已经在你自己的电脑上跑起来了，但**只有你这台电脑能访问**。你把这个地址发给别人，别人是打不开的，因为他们的浏览器里输入`127.0.0.1`访问的是他们自己的电脑，不是你那台。

> **核心区分**：`www.baidu.com` 是别人的电脑（互联网服务器），`127.0.0.1` 是你自己的电脑（本地服务器）。一个是“上网”，一个是“调试”。

### 2.2 网页生成的工作流程

你在地址栏敲下 `https://lib.njau.edu.cn` 然后回车，背后其实是这样的：

```
你输入网址
    ↓
浏览器问 DNS：“这个域名的服务器 IP 是啥？”
    ↓
DNS把服务器IP解析给浏览器，浏览器给服务器发请求：“把首页给我”
    ↓
服务器收到请求，开始处理
    ↓
服务器返回一堆数据（HTML + CSS + JS）
    ↓
浏览器拿到数据，渲染成你看到的页面
```



---

### 2.3 网页的三种类型

（1）静态页面：提前写好的HTML文件，一直存在服务器上。谁访问，什么时候访问内容都一模一样，改内容得改源文件。

（2）动态页面：用户访问的时候，服务器才临时生成网页。不同用户，不同时间看到的内容可能不一样，比如登录后的个人中心。  

（3）活动页面：把一部分计算工作放到用户的浏览器上做，既保留动态效果，又减轻服务器压力。

---

### 2.4 客户端和网页版的关系

同一个服务，往往有“网页版”和“客户端”两种打开方式。例如微信，你在浏览器里打开的是“网页版”，在手机上打开的是“App客户端”。

**它们共享了什么？**

客户端和网页版背后其实是同一套系统：

```text
网页版 ——→ 同一套后端 ←—— 客户端
                ↓
            同一套数据库
```

也就是说，你用网页版发了一条消息，客户端上也能看到——因为它们连的是同一个后端、同一个数据库。

**区别在哪？**

区别只在“前端”这一层：

|    对比项     |             网页版             |           客户端            |
| :--------: | :-------------------------: | :----------------------: |
|  **渲染方式**  | 浏览器解析 HTML / CSS / JS 来渲染界面 | 软件自己渲染界面，或在内部嵌入浏览器内核进行渲染 |
| **是否需要安装** |       不需要安装，打开浏览器即可使用       |     需要下载安装包或通过应用商店安装     |
|  **更新方式**  |        刷新浏览器即可使用最新版本        |     需要下载更新包或重新安装新版本      |
|  **访问方式**  |         通过 URL 地址访问         |      通过桌面图标或应用列表打开       |

所以客户端只是访问同一个系统的**另一种入口**。它不存储你的数据，也不处理你的业务逻辑——这些都交给服务器和数据库去做，客户端只管展示和交互。

---

### 2.5 单机软件和联网软件对比

和Web系统相对的概念是“单机软件”。比如植物大战僵尸、本地记账软件、其他单机游戏——它们都属于这一类。

**它的结构是：**

```text
程序
  ↓
本地数据库 / 本地文件
```
程序和数据都在你自己的电脑上，**不需要联网也能运行**。

**它的特点是：每个人的数据互不影响**

你在自己电脑上玩的植物大战僵尸存档，和你舍友电脑上的存档是完全独立的两个文件，没有任何关联。

而Web系统（比如知网、南农教务系统）则是所有用户共享同一份数据——你上传的论文，别人也能搜到。

## 3. Web 系统的三大核心板块

任何一个正经的 Web 系统，无论多复杂，扒开看都是三层：

```
         用户（你）
            ↓
    ┌───────────────┐
    │    前端        │  ← 你看到的界面、按钮、输入框
    │  （展示层）     │
    └───────────────┘
            ↓  （通过 HTTP 来回传数据）
    ┌───────────────┐
    │    后端        │  ← 处理逻辑：验证密码、查论文、算推荐
    │  （业务层）     │
    └───────────────┘
            ↓  （读写数据）
    ┌───────────────┐
    │   数据库       │  ← 真正存数据的地方：用户表、论文表
    │  （存储层）     │
    └───────────────┘
```

用户的操作永远从“前端”出发，经过“后端”处理，最终落到“数据库”。返回时原路折返：数据库把结果还给后端，后端加工后还给前端，前端展示给你看。整条链路是 **“请求往下走，响应往上走”**。

- **前端**：门面 + 服务员，负责接待你、记下你的需求。
- **后端**：后厨，负责炒菜、算账、做决策。
- **数据库**：冰箱 + 储物柜，负责长期存食材（数据）。

**举个南农的例子**——你在农业文献检索系统里搜“智慧农业”：

1. 前端把你输入的“智慧农业”打包成 HTTP 请求，发给后端。
2. 后端接到请求，去数据库里查所有标题含“智慧农业”的论文。
3. 数据库把匹配的论文记录返回给后端。
4. 后端把数据整理成 JSON 格式，返回给前端。
5. 前端把 JSON 渲染成漂亮的论文列表，展示给你。

**为什么要拆成三层？**

早期的 Web 应用经常把前端、后端、数据库的代码混在一起写（比如在 HTML 里直接写 SQL 查询），这样做小项目还行，但项目一大就乱成一锅粥——改一个地方可能要翻几十个文件，而且一个人改错了整锅端。

所以现在大家约定：**各管一摊，互不干扰**。前端只管展示和交互，后端只管业务逻辑和数据加工，数据库只管存数据。这样改前端不会影响后端，换数据库也不会影响前端——各司其职，耦合度低，维护起来轻松很多。

## 4. 前端基础（HTML/CSS/JavaScript）

前端有三个基本语言，分工很明确：

|       语言       |          作用          |     举例理解      |
| :------------: | :------------------: | :-----------: |
|    **HTML**    |  描述页面有什么（标题、图片、按钮）   | 房子的结构（承重墙、房梁） |
|    **CSS**     | 描述这些东西长什么样（颜色、大小、位置） | 装修（墙纸、灯光、家具）  |
| **JavaScript** | 描述这些东西怎么动（点击、弹窗、取数据） | 智能家居（门铃响了亮灯）  |

### 4.1 HTML——骨架

#### 4.1.1 超文本与HTML的本质定义

（1）超文本  
普通文本是线性的，你必须按照顺序从第一行读到最后一行。而超文本是“带跳转的文本”，点击页面里的蓝色链接，就能从一个文档跳转到另一个文档，这就是非线性的核心含义。平时我们上网的本质就是在超文本的信息节点之间不断跳转。  

（2）HTML  
用于描述Web文档结构与语义的描述性标记语言，是Web页面内容的载体与标准化表示形式，HTML文档为纯文本格式，所以你用文本编辑器就能写这个html文件代码，比如使用记事本，然后只需要把后缀改为.html即可。文本交由浏览器负责解析并渲染为可视化的Web页面。HTML不是编程语言，它本身没有逻辑判断，循环，变量这些编程能力，它唯一的作用是给内容打“标记”，目的是告诉浏览器这段文字是“段落”、“图片”、“这里有个标题”、“那里有个按钮”等。

---

#### 4.1.2 HTML文档的标准基本结构

标准HTML文档由文档头（Head）和文档体（Body）两个逻辑部分构成，通过根标记`<html>`
包裹形成完整的文档树。虽然现代浏览器很宽容（没有`<html>`也能显示）。但是还是得按标准来是不是？

```html
<html>
  <head>
    <!-- 文档头内容 -->
  </head>
  <body>
    <!-- 文档体内容 -->
  </body>
</html>
```

（1） 根标记`<html>` 
标识整个HTML文档的作用域，所有HTML内容必须包裹在`<html>`（起始标记）和`</html>`（结束标记）之间。

（2） 文档头标记`<head>`  
文档的元数据（Metadata）容器，用于存储页面的配置信息，引用资源与语义说明。其内容不会直接渲染在页面的可视化区域。也就是用户看不到 ，但是决定了存储页面的配置信息。  

`<head>`里常见的内容：  

- `<title>`：页面标题，显示在浏览器标签栏  
- `<meta>`：元信息，比如页面编码，关键词，描述  
- `<link>`：引用外部CSS文件  
- `<script>`：引用JavaScript代码

（3） 文档体标记`<body>`  
文档的内容容器，所有需要呈现给用户的可视化内容（文本，图片，链接，表格，表单等）必须放置在`<body>`与`</body>`之间。

---

#### 4.1.3 **HTML标记的语法规范与分类**

HTML标记是用于标识内容语义的语法单元，必须使用<>包裹，标记名称不区分大小写，但推荐统一使用小写。

 **标记的两大类**

（1） 双标记（容器标记）  
由起始标记和结束标记成对构成的语法单元 。
语法格式：<标记名> 被标记的内容 </标记名>  

>示例：`<title>页面标题</title>`  

（2） 单标记（空标记）  
无需包裹内容，单独使用即可表达完整语义的标记，用于表达独立的，无内容的页面元素。  
语法格式：<标记名>  

>示例:`<br>（换行）、<hr>（水平线）、<img>（图片）、<input>（输入框）`


 **标记属性的语法**

属性是用于拓展标记功能，配置标记特性的语法单元，通过键值对的形式写在起始标记内部，用于为标记提供额外的语义信息或显示配置。

标准语法格式：
  `<标记名 属性1="属性值" 属性2="属性值" ...>`

示例：

```html
   <a href="https://www.njau.edu.cn" target="_blank">南农官网</a>
```

属性与标记的绑定关系：
>[!tip]
>（1）属性不能脱离标记单独存在，每个标记有自己专属的属性集，也有所有标记通用的全局
>	属性。
  （2） 引号规范：属性值推荐使用双引号包裹，单引号也兼容。
  （3） 布尔属性：部分属性只有“开启/关闭”两种状态，写属性名就表示开启，不写就表示关闭，不需要额外赋值，这类就是bool属性。

---

#### 4.1.4 HTML注释的语法

注释是HTML文档中的说明性内容，浏览器解析时会完全忽略注释内容，不会将其渲染到页面中。

标准语法：

```
<!-- 这是单行注释 -->

<!--
这是多行注释
可以写很多行
-->
```

#### 4.1.5 HTML常见标签

- `<h1>` ~ `<h6>`：标题，数字越小等级越高
- `<p>`：段落，放普通文字
- `<img>`：图片，需要 `src` 属性告诉地址
- `<a>`：超链接，点它跳转
- `<div>`：分区容器，用于将多个元素组合为一个逻辑区块，实现页面的模块化布局
- `<input>`：输入框，让用户打字
- `<button>`：按钮，用来点击
- `<form>`：表单，包住一堆输入框，点提交一起发出去

---

#### 4.1.6 三类列表标签

列表是HTML中结构化展示内容的核心方式，分为无序列表，有序列表，定义列表三类。列表支持嵌套使用。嵌套时记住一条铁律：**标签必须严格按层级开闭，不能交叉**。比如 `<ul><li>` 可以，`<ul><li></ul>` 不行。

（1）无序列表`<ul> ` 
用于展示无先后顺序的并列内容，每个列表前显示项目符号，通过type属性配置符号样式。
```html
<ul type="disc|circle|square">
   <li>第一步</li>
   <li>第二步</li>
   <li>第三步</li>
</ul>
```

（2） 有序列表`<ol>`  
用于展示有先后顺序，优先级的内容，每个列表项前显示序号，通过type属性配置序号类型。

```html
<ol type="1|A|a|I|i">
   <li>第一步</li>
   <li>第二步</li>
   <li>第三步</li>
</ol>
```

（3） 定义列表`<dl>`  
用于展示“术语-解释”的键值对结构，适合名词解释，产品参数，问答等场景。

```html
<dl>
   <dt>术语1</dt>
   <dd>术语1的解释说明</dd>
   <dt>术语2</dt>
   <dd>术语2的解释说明</dd>
</dl>
```
- `<dt>`：定义术语（Definition Term）
- `<dd>`：定义描述（Definition Description）

---

#### 4.1.7 超链接——Web 的灵魂

`<a>` 标签是 HTML 里最重要的标签之一，它实现了 Web 最核心的特性——**从一个页面跳转到另一个页面**。
```html
<a href="目标地址" target="打开方式">链接显示的文字</a>
```

|  场景  |               写法示例                |             说明             |
| :--: | :-------------------------------: | :------------------------: |
| 外部链接 |  `href="http://www.njau.edu.cn"`  | 必须带 `http://` 或 `https://` |
| 本地页面 |        `href="about.html"`        |         同目录下直接写文件名         |
| 页内锚点 |           `href="#top"`           | 跳转到当前页面里 `name="top"` 的位置  |
| 文件下载 |        `href="course.zip"`        |      浏览器会自动下载，不会打开文件       |
| 发邮件  | `href="mailto:admin@njau.edu.cn"` |       点击自动则打开默认邮件客户端       |

>**target 属性**：控制链接在哪里打开，也就是打开方式，。最常用的是 `_blank`（新窗口）和 `_self`（当前窗口，默认）。

---

#### 4.1.8 表格——展示结构化数据

表格是HTML中展示结构化数据，实现传统布局的核心标签。

表格为二维结构化容器，由行，单元格，表头构成，遵循“表格-->行-->单元格”的三级嵌套层级结构。表头其实就是特殊的单元格，其他的普通单元格存的是数据。

```html
<table border="1">
  <caption>论文列表</caption>
  <tr>
    <th>标题</th>   <!-- th 是表头，自动加粗居中 -->
    <th>作者</th>
  </tr>
  <tr>
    <td>农业数字化研究</td>  <!-- td 是普通单元格 -->
    <td>张三</td>
  </tr>
</table>
```

核心标签说明：
- `<table>`：表格容器
- `<caption>`：表格标题，默认居中显示在表格上方
- `<tr>`：表格行（Table Row），每一个`<tr>`代表一行
- `<th>`：表头单元格（Table Header）
- `<td>`：普通单元格（Table Data）

##### 4.1.8.1 **跨行跨列（合并单元格）**：

**语法**：
（1） 跨列合并（左右合并）：colspan="合并的列数"
示例：`<td colspan="2">这个单元格占2列</td>
`
（2） 跨行合并（上下合并）：rowspan="合并的行数"
示例：`<td rowspan="3">这个单元格占3行</td>`

##### 4.1.8.2 **表格分组**

大型表格可以分成表头、表体、表尾三部分，方便单独设置样式：  

```html
<table>
 <thead> <!-- 表头部分 -->
    <tr><th>表头</th></tr>
</thead>
 <tbody> <!-- 表体部分，可以有多个 -->
    <tr><td>内容</td></tr>
 </tbody>
<tfoot> <!-- 表尾部分 -->
    <tr><td>合计</td></tr>
  </tfoot>
</table>
```

---

#### 4.1.9 图片——嵌入图像

**图片基础格式**
>（1）GIF：最多256色，文件小，适合图标，小动画。  
>（2）JPEG：真彩色（约1600万色），适合照片，彩色大图，但他是有损压缩，不适合存图标、>文字、logo，因为会变模糊。
>（3）PNG：真彩色，适合图标、logo，缺点是文件比JPEG大。
>（4）Webp：谷歌推出的现代格式，同等画质下文件比JPEG和PNG都小，而且同时支持有损/无损压缩和透明。现在主流浏览器都支持了，能用的场景尽量用。


图片标签为单标记，用于在页面中嵌入外部图像资源，属于行内元素，支持通过属性控制尺寸，布局，替代文本。

```html
语法：  
<img
  src="图片路径"
  alt="图片加载失败时显示的文字"
  width="宽度"
  height="高度"
  border="边框宽度"
  hspace="水平边距"
  vspace="垂直边距"
  align="对齐方式"
>
```

核心属性
>[!tip]
>（1） src：必填，图片的路径，支持相对路径（本地地址）和网络 URL（网络地址）
>（2） alt：必填，替代文本，图片加载失败时显示，也是盲人读屏软件读取的内容，是 Web 无障碍的核心要求
>（3） width/height：图片尺寸，只写一个会等比例缩放，两个都写可能拉伸变形
>（4） align：图文混排核心属性(垂直对齐，环绕对齐)

---

#### 4.1.10 表单——Web交互的核心

表单是HTML提供的标准化交互组件集合，用于收集用户输入的数据，并通过HTTP协议将数据提交给服务器端处理，是Web实现“用户输入--服务器处理--结果返回”交互模式的核心载体。是所有网站登录，出册，搜索，提交评论，上传文件功能的基础，没有表单就没有交互式网站。

##### 4.1.10.1 **表单容器`<FORM>`**

所有表单元素必须放在`<FROM>`标签内
```html
<form action="处理地址" method="get|post">

<!-- 所有输入框、按钮都写在这里 -->

</form>
```

>(1) action：表单数据提交到的服务器地址，可以是后端接口地址，也可以是邮箱（mailto:xxx@xx.com）
>(2) method：数据提交的 HTTP 方法（GET与POST）

| **对比项** |           **GET**            |       **POST**       |
| :-----: | :--------------------------: | :------------------: |
|  数据位置   | 拼接在 URL 后面，?name=xxx&pwd=xxx | 放在 HTTP 请求体里，URL 看不到 |
|  长度限制   |        有，URL 最长约 2KB         |         无限制          |
|   安全性   |         低，数据明文显示在地址栏         |       高，数据不可见        |
|  适用场景   |        搜索、查询、无敏感数据的提交        |  登录、注册、上传文件、敏感数据提交   |
|  服务器获取  | Request.QueryString["name"]  | Request.Form["name"] |

---

##### 4.1.10.2 表单核心输入元素

单标记，通过type属性实现不同的输入类型

（1） 文本输入与密码框
```html
<!-- 单行文本框 -->
 用户名：<input type="text" name="username" size="20" maxlength="16">
<!-- 密码框，输入内容显示为圆点 -->
 密码：<input type="password" name="password" size="20" maxlength="16">
```

-  name：必填，服务器获取数据的标识，没有 name 的输入框数据提交不了
-  size：输入框的显示宽度（字符数）
-  maxlength：允许输入的最大字符数

（2） 单选按钮与复选框
```html
<!-- 单选按钮：同一组name必须相同，才能实现互斥 -->
 性别：
<input type="radio" name="gender" value="male" checked> 男
<input type="radio" name="gender" value="female"> 女
<!-- 复选框：同一组name可以相同，支持多选 -->
爱好：
<input type="checkbox" name="hobby" value="read" checked> 读书
<input type="checkbox" name="hobby" value="sport"> 运动
<input type="checkbox" name="hobby" value="music"> 音乐
```

（3） 提交与重置按钮
```html
<input type="submit" value="提交表单">
<input type="reset" value="重置填写">
```

- submit：点击后自动提交整个表单的数据到 action 地址
- reset：点击后清空所有输入框，恢复到默认值

（4）隐藏域与图片按钮  

- 隐藏域：
```html
<input type="hidden" name="id" value="123">
```
用户看不到，用来传递不需要用户输入的参数

- 图片按钮：
```html
<input type="image" src="submit.jpg">
```
用图片当提交按钮

（5）下拉选择框

用于选项较多的场景，比单选按钮更节省空间，分弹出式和列表式两种

```html
<!-- 弹出式（默认，size=1） -->
学历：
<select name="education">
  <option value="high">高中</option>
  <option value="college" selected>本科</option>
  <option value="master">硕士</option>
</select>
<!-- 列表式（size>1，显示多行） -->
<select name="city" size="3" multiple>
  <option value="bj">北京</option>
  <option value="sh">上海</option>
  <option value="gz">广州</option>
  <option value="sz">深圳</option>
</select>
```
- size：显示的选项数量，大于 1 就是列表式
- multiple：允许多选（按住 Ctrl 选多个）
- selected：默认选中的选项

（6） 多行文本框

用于输入大段文字，比如留言，评论，简介

个人简介：
```html
<textarea name="intro" rows="5" cols="50">
这里是默认内容
</textarea>
```
- rows：显示的行数
- cols：显示的列数（每行的字符数）

#### 4.1.11  音视频——嵌入音频与视频

（1）音频标签`<audio>`
```html
<audio controls autoplay loop>
  <source src="music.mp3" type="audio/mpeg">
  您的浏览器不支持音频播放
</audio>
```
- controls：显示播放、暂停、音量控制条
- autoplay：自动播放（现代浏览器大多禁止自动播放，需要用户交互）
- loop：循环播放

（2）视频标签`<video>`
```html
<video width="600" height="400" controls poster="cover.jpg">
  <source src="video.mp4" type="video/mp4">
  您的浏览器不支持视频播放
</video>
```
- poster：视频加载前显示的封面图
- muted：静音播放

#### 4.1.12 **页面自动刷新与跳转**

通过 META 标签实现，写在`<head>`里

```html
<!-- 5秒后自动跳转到首页 -->
<meta http-equiv="refresh" content="5; url=index.html">
<!-- 每30秒自动刷新当前页面 -->
<meta http-equiv="refresh" content="30">
```

>常用于登录成功跳转、考试系统倒计时、实时数据页面刷新

### 4.2 CSS——样式

#### 4.2.1 CSS的本质定义与价值

**CSS的本质：**

（1）CSS 负责让页面变好看。你不用死记硬背所有属性名，**但你需要知道“CSS能控制哪些方面的样式”**——颜色、大小、位置、动画。具体怎么写，让AI生成就好。

（2） **CSS也是一个纯文本文件**，和HTML一模一样。你用记事本、VS Code打开它，里面全是字母和符号，没有任何神秘的东西。

（3）**CSS**的文件名后缀是 `.css`，里面写的是规则，比如 `p { color: red; }`，作用是告诉浏览器“给段落上红色”。

（4）你可以把CSS理解为**一份“装修说明书”**——它本身不是房子（不是页面内容），它是一份独立于房子之外的文档，专门告诉浏览器“这个区域的墙刷什么颜色、那扇门多大尺寸”。

（5）CSS和HTML代码一样，都是是样式语言：没有逻辑判断，没有循环，它就是一条一条的“给某某元素把某某属性改成某某值”的指令。它像“装修清单”——不会思考，只会照着清单执行。

**现代网页布局主要靠两种CSS工具 ：**

>（1） **Flexbox**：管"一排东西怎么对齐"。比如导航栏里的"首页、论文库、个人中心"三个链接，你要它们水平均匀分布、垂直居中——Flexbox一句话搞定。
>
>（2）**Grid**：管"整个页面怎么切块"。比如左菜单、右内容、上导航、下页脚这种四块结构——Grid一刀切成棋盘，各自填内容就行。

实际开发中两者通常搭配使用：**外层用Grid切大块，里层用Flexbox排细节**——就像盖房子，Grid是户型图（几室几厅），Flexbox是家具摆放（沙发茶几怎么对齐）。


**CSS的核心价值**  

>（1） 一次定义，全站生效：写一个CSS规则，整个网站所有对应元素都生效，改一次全网站同步更新。  
>（2） 精准控制所有样式：小到一个字的颜色，大到整个页面的布局，像素级精确控制。  
>（3） 跨平台一致显示：解决了不同浏览器显示不一样的问题，所有浏览器都遵循CSS标准。

---

#### 4.2.2 四种引入使用CSS的方式（优先级是核心）

|  引入方式   |                 写法位置                  |      作用范围      | 优先级（从高到低） |      适用场景      |
| :-----: | :-----------------------------------: | :------------: | :-------: | :------------: |
|  内嵌样式   |        写在 HTML 标签的 `style` 属性里        |  只作用于当前这一个标签   |   1（最高）   |   单个元素临时改样式    |
|  内部样式表  |       写在 HTML 头部的 `<style>` 标签里       |  只作用于当前这一个页面   |     2     |   单个页面的专属样式    |
| 链入外部样式表 |    单独写在 `.css` 文件里，用 `<link>` 标签引入    | 所有引入这个文件的页面都生效 |     3     | 整个网站的公共样式（最常用） |
| 导入外部样式表 | 在 `<style>` 里用 `@import` 导入 `.css` 文件 |   所有引入的页面生效    |   4（最低）   |  现在基本不用，了解就行   |

**一句话解释：**
（1）内嵌样式：**打补丁式装修，哪儿坏贴哪儿，只适合救急。**
（2）内部样式表：**给这一页装了一份装修清单，这一页有效，下一页还要重新写。**
（3）链入外部样式表：**全公司共用一份装修手册，手册改了，所有办公室一起变样。最常用。**

**性质：**
- 越靠近元素的样式优先级越高，后定义的样式覆盖先定义的样式。
- 内嵌 > 内部 > 外部，同一个元素多个样式冲突时，永远听优先级高的。CSS的优先级遵循“**就近原则**”

**我的习惯是**：

写课程设计或小项目时，直接用一个外部`.css`文件链入所有页面，维护最省事。行内样式只在调试时临时用（比如在浏览器F12里试颜色），正式代码里基本不写——因为它优先级最高，后期改起来最麻烦。

---

#### 4.2.3 CSS的核心语法：选择器

CSS 的语法很简单：选哪个元素 { 改什么属性: 改成什么值; }
整个 CSS 的核心就是「怎么准确选中你要改的元素」—— 也就是选择器


 **五类核心选择器**

|    选择器     |        写法        |              作用               |  优先级   |
| :--------: | :--------------: | :---------------------------: | :----: |
| **ID 选择器** |    `#nav { }`    | 选中页面里**唯一的一个**元素（一个 ID 只能用一次） |   最高   |
|  **类选择器**  |    `.btn { }`    | 选中所有加了 `class="btn"` 的元素（最常用） |   高    |
| **标签选择器**  |     `p { }`      |     选中页面里**所有** `<p>` 标签      |   中    |
| **包含选择器**  |   `.nav a { }`   | 选中 `.nav` **里面**的 `<a>`，不影响外面 |   低    |
|  **选择器组**  | `h1, h2, h3 { }` |        同时给多个选择器加一样的样式         | 同各自优先级 |

> 💡 **优先级口诀**：ID > 类 > 标签。同一个元素被多个规则命中时，优先级高的生效。

---

#### 4.2.4 **伪类：给元素加状态样式**

伪类是 CSS 最实用的功能，核心作用是：给元素的不同状态加不同的样式。

 **最经典、最常用、必考的就是链接的四个状态伪类**
 >[!note]
（1） a:link：链接没被点击过的默认样式
（2） a:visited：链接已经被访问过的样式
（3） a:hover：鼠标悬停在链接上的样式
（4） a:active：鼠标点击链接那一瞬间的样式  
注：顺序绝对不能乱，顺序乱了样式将不再生效

例如：
```html
a:link   { color: blue; }   /* 没访问过，链接呈现蓝色 */
a:visited { color: purple; } /* 访问过了，链接呈现紫色 */
a:hover  { color: red; }    /* 鼠标悬停（最常用），链接呈现红色 */
a:active { color: green; }  /* 点击的瞬间，链接呈现绿色 */
```

**补充**：严格来说，这4个里有**2个**是链接专属的（`:link`和`:visited`只能用在`<a>`标签上），但`:hover`和`:active`是所有元素都能用的——按钮、卡片、表格行、图片，任何东西都能用。

---
#### 4.2.5 CSS滤镜：给元素加特效

CSS 滤镜就是给图片、文字加视觉特效，你不需要记任何参数，只要知道滤镜能实现什么效果就行，需要的时候直接告诉 AI：

>（1） 模糊效果：图片、文字变模糊
>（2） 透明度：调整元素透明程度
>（3） 灰度：把彩色图片变成黑白
>（4） 反色：颜色反转
>（5） 阴影：给文字、图片加阴影
>（6） 发光：给元素边缘加发光效果
>（7） 波纹：给图片加水波纹扭曲效果
>（8） 渐变：给背景加渐变色（从一个颜色过渡到另一个颜色）
>（9） 动态过渡：鼠标悬停时平滑变化（不是突然变，是慢慢变）

还有老的 IE 转换滤镜（百叶窗、开门、渐隐、螺旋等页面切换效果），现在基本不用了，了解就行。

### 4.3 JavaScript——交互

#### 4.3.1 JavaScript本质

JavaScript是一种专门跑在浏览器里的脚本语言。它的工作方式是：用户做了某个操作（比如点击按钮），就触发对应的代码去执行。它不能像C++那样读写你电脑上的文件，所以相对安全。它通常写在HTML文件里，和HTML配合工作。

JavaScript（简称 JS）是网站的「交互灵魂」—— 没有 JS，你的网站就是只能看、不能动的静态页面；有了 JS，才能实现点击弹窗、表单验证、下拉菜单、轮播图、动态加载这些所有你能想到的「点了有反应」的功能。

JS的特点：

>（1） 运行在浏览器里：不用服务器，用户打开网页的时候，JS 就在他自己的电脑上运行，速度快
>（2） 事件驱动：用户做什么动作（点击、鼠标移上去、输入文字），就触发对应的 JS 代码
>（3） 动态修改页面：不用刷新页面，就能直接改页面上的文字、图片、样式

---

#### 4.3.2 JavaScript与C++对比

**JavaScript是编程语言。而且是目前世界上使用范围最广的编程语言。**

|    对比项     |         JavaScript         |            C++             |
| :--------: | :------------------------: | :------------------------: |
| **在哪儿运行**  | 原本只在浏览器里，现在Node.js也能在服务器上跑 |  直接跑在你的电脑操作系统上（编译成.exe文件）  |
| **要不要编译**  |      不用编译，浏览器直接读源代码执行      |       必须先编译成机器码才能运行        |
|  **能干啥**   |      操控网页、做交互、弹窗、请求数据      |    写操作系统、游戏引擎、大型软件、驱动程序    |
| **访问电脑硬件** | 基本不行（浏览器出于安全限制，不让你读写硬盘文件）  | 可以，C++能直接操作内存、读写文件、调用系统API |
|   **难度**   |          相对友好，上手快          |   复杂，要手动管理内存，指针、引用等概念挺难的   |

一句话解释：
- **C++** = 自己盖房子。你要自己拌水泥、砌砖、拉电线，每一个细节都要亲手掌控。能做摩天大楼，但累死人，盖错了整栋楼可能塌。

- **JavaScript** = 住精装公寓。你不用管地基和承重墙，你只需要按开关、调灯光、布置家具。干不了重活，但日常住着绰绰有余。

---

#### 4.3.3 JavaScript和前端两个文件类型（HTML / CSS）怎么联动？

一句话解释：
**HTML是骨架，CSS是皮肤，JavaScript是神经系统。三者各干各的，但缺一个你都动不了。**

具体怎么联动：

**第一步：HTML打底子**

```html
<!DOCTYPE html>
<html>
<head>
  <title>我的界面</title>
  <link rel="stylesheet" href="style.css">  <!-- 关联CSS文件 -->
</head>
<body>
  <button id="myBtn">点我</button>           <!-- 一个按钮 -->
  <p id="msg">你好</p>                       <!-- 一段文字 -->
  
  <script src="script.js"></script>         <!-- 关联JS文件（放在最下面） -->
</body>
</html>
```

此时页面长这样：一个按钮，一段文字"你好"。只有样子，点按钮不会有任何反应。

**第二步：CSS美化**

```css
/* style.css */
button {
  background-color: blue;
  color: white;
  padding: 10px 20px;
  border: none;
  border-radius: 5px;
}
```

按钮变蓝底白字、圆角、内边距——看起来更像"能点的东西"了。但点它还是没反应。

**第三步：JS让它动起来**

```javascript
// script.js
document.getElementById("myBtn").onclick = function() {
  document.getElementById("msg").textContent = "点的人是gay！";
};
```

现在你点按钮，文字从"你好"变成"点的人是gay！"——这就是JavaScript干的活。**它找到了HTML里的元素（`getElementById`），监听用户动作（`onclick`），然后修改内容（`textContent`）。**


**联动方式**：HTML用`<link>`引用CSS，用`<script>`引用JS。浏览器依次读这三个文件，合在一起渲染出你能看到的"网页"。

---

#### 4.3.4 **JS 能实现什么功能**

1. **表单验证**：用户填注册表单的时候，实时判断密码够不够长、邮箱格式对不对，不用提交到服务器就能提示
2. **弹窗提示**：点击按钮弹出提示框、确认框
3. **动态效果**：轮播图、下拉菜单、折叠面板、标签页切换
4. **页面交互**：点击按钮改文字、换图片、显示 / 隐藏某个区域
5. **数据处理**：简单的计算、排序、筛选
6. **异步加载**：不用刷新整个页面，只更新页面的一部分（比如加载更多新闻）

---

#### 4.3.5 **JS 的三大核心机制**

##### (1) **事件驱动：用户做动作，JS 才执行**

JS 不会自己跑，永远是「用户触发了某个事件，才执行对应的代码。

最常用的事件

>（1） onclick：用户点击了某个元素（按钮、图片、链接）
>（2） onmouseover：鼠标移到了元素上面
>（3） onmouseout：鼠标从元素上移开
>（4） onload：页面加载完成
>（5） onchange：输入框、下拉框的内容变了
>（6） onsubmit：表单提交的时候

举个例子：你点击按钮弹出提示框 —— 本质就是「按钮的onclick事件，触发了「弹出提示框」的 JS 代码」。

##### (2) **DOM 操作：JS 能随便改页面**

你写的 HTML 是一堆文本，但浏览器不认文本，它只认对象。所以浏览器把你的 HTML 读进去之后，在内存里建了一棵“树”，这棵树就是 DOM。**DOM 就是浏览器给 JS 提供的“操作网页的官方接口”——没有这个接口，JS 对着网页就是个瞎子。**

例如：

```html
<html>
  <head><title>我的页面</title></head>
  <body>
    <div id="box">
      <p>Hello</p>
      <a href="#">链接</a>
    </div>
  </body>
</html>
```

浏览器在内存里把它变成这棵树：

```text
document（根节点）
  └── html
        ├── head
        │     └── title
        │           └── "我的页面"
        └── body
              └── div（#box）
                    ├── p
                    │     └── "Hello"
                    └── a
                          ├── href="#"
                          └── "链接"
```

**每个尖括号标签都变成一个“节点”，标签里的文字变成“文本节点”，标签之间的关系（父子、兄弟）也在这棵树上原样保留。**

JS 通过 DOM 能干什么：

>（1） 改文字：把按钮上的「提交」改成「提交中」
>（2） 改样式：点击按钮让某个 div 显示 / 隐藏、变颜色
>（3） 加元素：动态往页面里加一行新的内容
>（4） 取值：拿到用户在输入框里写的内容
>（5） 删元素： 从树上摘掉这个节点，页面上对应元素消失

这就是 JS 的核心能力：**JS 从来不直接操作“页面显示”本身，JS 只是操作这棵树。树变了，浏览器自动重新渲染页面。**

##### (3) **核心对象（JS 的工具库）**

（1） windows对象：代表浏览器窗口，最常用的功能：  
-  alert()：弹出提示框
-  confirm()：弹出确认框（确定 / 取消）
-  open()：打开新窗口

（2） document 对象：代表整个 HTML 页面，最常用的功能：
-  找到页面里的某个元素
-  修改元素的内容、样式

（3） 表单对象：拿到用户在表单里输入的内容，验证格式

（4） 内置对象:Math(数学计算)，Date（日期计算），Array（数组）

---

#### 4.3.6 JS的三种引入方式

和 CSS 一样，JS 也有三种写法

（1） 内部 JS：写在 HTML 的`<script>`标签里，只作用于当前页面

（2） 外部 JS：单独写在.js文件里，多个页面可以引入，适合公共功能（比如所有页面都要用的表单验证）

（3） 行内 JS：直接写在 HTML 标签的事件里，比如`<button onclick="alert('点我了')">`，只适合单个简单功能

## 5. 现代前端框架——为什么不用纯手写

如果页面就两三个按钮，直接用 HTML/CSS/JS 没问题。但要做一整个系统（比如南农的论文管理平台），页面几十个、组件到处复用、数据到处窜，手写会累死人。这时候就该上 **前端框架** 了。

### 5.1 框架解决啥问题

- **组件化**：把页面拆成一块块积木（导航栏、搜索框、论文卡片），每块独立开发、独立维护。
- **数据驱动**：数据变了，页面自动更新，不用手动操作 DOM。
- **路由管理**：不同 URL（网址）对应不同页面，切换时不刷新浏览器，体验更流畅（传统网站点链接会整个页面刷新，白屏一下）。
- **工程化**：项目结构清晰，代码好维护。

---

### 5.2 主流前端框架怎么选

|     框架      |      特点       |            适合场景            |
| :---------: | :-----------: | :------------------------: |
|   **Vue**   | 简单、中文文档友好、上手快 | 中小型系统、管理系统、科研平台（**推荐初学者**） |
|  **React**  | 生态庞大、灵活、大厂在用  |           大型复杂应用           |
| **Angular** |    重量级、全能     |     企业级大型项目（不适合初学者上手）      |

对于我们信息与计算科学/信息资源管理这类非纯计算机专业来说，**主攻 Vue** 就够用，配合 Python 后端即可，足够覆盖课程设计、科研项目、毕业设计等绝大多数场景。如果你是小项目或个人开发，看到这里就够了，Vue和React适合更复杂的场景，我自己在毕设里用的是第10章那套，具体原因在第10章有讲。React 不是不需要，等真有大型复杂项目需求再去学也不迟。

---

### 5.3 组件化思想（重中之重）

组件就是页面的“零件”。比如论文系统首页可以拆成：

```
首页
├── 导航栏（Navbar 组件）
├── 搜索框（SearchBox 组件）
├── 论文列表（PaperList 组件）
│   └── 单篇论文卡片（PaperCard 组件，循环渲染）
└── 分页（Pagination 组件）
```

每个组件有自己的模板、样式和逻辑，互不干扰。改导航栏的颜色，不会影响到论文卡片。

---
### 5.4 项目目录长啥样（以 Vue 3 为例）

```
my-paper-system/
├── src/
│   ├── components/   → 小组件（SearchBox.vue, PaperCard.vue）
│   ├── views/        → 页面（Home.vue, Login.vue）
│   ├── router/       → 路由配置
│   ├── stores/       → 状态管理（存登录用户信息）
│   ├── assets/       → 图片、公共样式
│   ├── App.vue       → 根组件
│   └── main.js       → 入口文件
├── package.json      → 依赖清单
└── README.md
```

## 6. 后端基础——服务器在忙什么

前端是“门面”，后端就是“后台办公室”——用户看不见，但所有正经事都是它干的。

### 6.1 后端到底干些啥

四件大事：
1. **处理业务逻辑**：比如“普通用户只能看，管理员才能删”。
2. **操作数据库**：接到查询就去捞数据，接到新增就往里插。
3. **用户认证**：判断你是谁、你能干什么。
4. **提供 API 接口**：向外暴露功能入口，让前端调用。

### 6.2 后端语言和框架怎么选

|       语言 + 框架        |       特点        |        适合场景         |
| :------------------: | :-------------: | :-----------------: |
| **Python + FastAPI** | 快、自动生成文档、天然支持异步 | 科研系统、AI 项目（**最推荐**） |
|    Python + Flask    |      轻量、灵活      |      简单应用、原型开发      |
|  Java + Spring Boot  |     稳定、性能强      |       大型企业系统        |
|  Node.js + Express   |    前后端都写 JS     |      全栈 JS 项目       |

信科/信管 + 科研方向，**Python + FastAPI** 是最佳拍档。不过如果你跟我一样，是从 Flask 开始接触 Web 开发的，也不用急着切换——Flask 足以跑通课程设计和毕设，等你需要写自动生成文档的 API 时再切 FastAPI 也不晚。

### 6.4 动态网页技术演进

#### 6.4.1 静态网页 vs 动态网页

这是 Web 发展史上最重要的分水岭：

|          |       静态网页       |                动态网页                |
| :------: | :--------------: | :--------------------------------: |
|    内容    |   所有人看到的内容都一样    |         不同用户、不同时间可能看到不同内容          |
|   运行位置   | 浏览器直接解析 HTML 文件  |     服务器先运行程序，生成 HTML 后再发送给浏览器      |
| 能不能连接数据库 |       ❌ 不能       |                ✅ 能                 |
|   常见后缀   | `.html` / `.htm` | `.asp` / `.php` / `.jsp` / `.aspx` |

**静态网页**：你写好一个 `.html` 文件放服务器上，谁打开都一样，服务器直接把文件发过去。

**动态网页**：你写的是“模板”，用户请求时服务器先跑代码、查数据库，把数据填进去，生成一个**独一无二的 HTML** 再发给用户。你登录淘宝看到的是你的购物车，我登录看到的是我的购物车——这就是动态。

**活动页面**：把一部分计算工作放到用户的浏览器上做（也就是早期的JavaScript干的事），既保留动态效果，又减轻服务器压力。

| :-------------------------: | :-----: | :---------------: |
|             ASP             | Windows |  已淘汰，考试可能考，开发不用   |
|             PHP             |   全平台   |   仍广泛使用，中小型网站首选   |
|             JSP             |   全平台   |   逐渐减少，大型老系统在用    |
| [ASP.NET](https://asp.net/) | Windows |   微软生态在用，但不如从前    |
|  **Python + Django/Flask**  |   全平台   | **现在推荐**，开发快，适合科研 |
|   **Java + Spring Boot**    |   全平台   | **企业级首选**，稳定、性能强  |

现在实际开发中，ASP 已经没人用了，PHP 还在用但逐年减少，主流是 **Python 系（Django/Flask）** 和 **Java Spring Boot**。

### 6.5 Web 服务器与应用服务器


你把一个网站搞到公网上让别人访问，要面对两个实际问题：

（1）**流量压力**：万一几百个人同时访问，那个Flask自带的`werkzeug`开发服务器直接卡死。
（2）**职责分离**：有人要看图片、CSS、JS（静态资源），有人要查数据库（动态逻辑），混在一起效率极低。

所以生产环境里，大家不搞“一个进程干所有活”，而是**拆成两拨人干活**：

| 角色             | 谁干                           | 干什么                                                                             | PyCharm里的情况                                  |
| -------------- | ---------------------------- | ------------------------------------------------------------------------------- | -------------------------------------------- |
| **Web服务器（入口）** | Nginx / Apache               | 站在最前面接待客人。客人要图片（静态资源），它直接给；客人要查论文（动态请求），它把活儿转给后面的人。它不干重活，但接待能力强，能抗住成千上万的人同时来敲门。 | **Flask开发服务器包揽了这个活儿**（但它只能抗住几个人同时用）          |
| **应用服务器（业务）**  | Tomcat / FastAPI进程 / Node.js | 躲在后面干重活。接到转来的请求，去查数据库、算逻辑、拼数据，再把结果还给Web服务器。它不直接面对客户端（浏览器），只跟Web服务器说话。           | **Flask代码本身干的就是这个活儿**（但Flask开发服务器把它和自己混在一起了） |

**注意：你平时在PyCharm里点“运行”时，Flask自带的开发服务器（Werkzeug）为了让你调试方便，把“Web服务器”和“应用服务器”的角色合并成了“一个全能管家”。你当前不用区分这两个角色，因为那是部署到公网面临大流量时才需要考虑的事。等你真正需要上线让别人访问时（比如买云服务器部署），才会用到Nginx（Web服务器）+ Gunicorn/Uvicorn（应用服务器，跑你的Flask代码）这种拆分的架构。初学阶段，你知道有这两个概念即可，先用PyCharm的开发服务器跑通功能要紧。**

在传统教材中，常见“Web服务器”包括 Apache、Nginx 等。  
  
但在现代 Web 架构中，需要区分以下概念： 

**（1） Web Server（Web服务器）**  
  
主要负责：  
- 接收 HTTP 请求  
- 处理静态资源  
- 转发请求  
  
代表：  
- Apache HTTP Server  
- Nginx（现代主流）  
  
**（2）Application Server（应用服务器）**  

负责运行后端业务逻辑，例如：  
- Java Web应用  
- Servlet  
- Spring Boot 应用  
  
代表：  
- Tomcat  
- Jetty  
  
**（3） 现代主流架构**  
  
实际生产中通常组合使用：  

  Nginx（入口层）  
  ↓  
  Tomcat / FastAPI / Node.js（业务层）  
  ↓  
  数据库

### 6.6 三种服务端数据存储方式的对比（Cookie vs Session vs Application）

初学者最容易搞混这三样东西，因为它们都跟“存数据”有关，但**存的位置不同、作用范围不同、生命周期不同**。

|    对比项    |         **Cookie**         |      **Session**       |   **Application**    |
| :-------: | :------------------------: | :--------------------: | :------------------: |
|  **存在哪**  |          用户的浏览器里           |        服务器的内存里         |       服务器的内存里        |
| **谁能看到**  |   用户自己能看到（浏览器开发者工具里明文可见）   |     用户看不到，只有服务器能读到     |    用户看不到，只有服务器能读到    |
| **作用范围**  |         单个用户、单个网站          |   单个用户、整个网站（所有页面共享）    | **所有用户**、整个网站（全局共享）  |
| **生命周期**  |     可以设置过期时间，关掉浏览器可能还在     |     **默认关掉浏览器就失效**     |   **服务器不重启就一直有效**    |
| **存什么数据** | 少量不重要的数据：记住用户名、语言偏好、购物车商品数 | 重要敏感数据：用户 ID、登录状态、权限角色 | 公共数据：网站访问量、在线人数、系统公告 |
|  **安全性**  |      低（存在用户浏览器，可被篡改）       |     高（存在服务器，用户动不了）     |    高（存在服务器，用户动不了）    |
| **容量限制**  |        小（一般 4KB 左右）        |     较大（取决于服务器内存配置）     |    较大（取决于服务器内存配置）    |

通俗易懂的讲：
- **Cookie** = 你自己钱包里放的会员卡。卡上写着你的名字和积分，你自己能看到也能改，但改完可能刷不出来（因为服务器那边会对不上）。

- **Session** = 商场的 VIP 档案柜。你报手机号，工作人员从柜子里拿出你的专属档案，里面有你的消费记录和权限。你走了档案就收回去，别人拿不走也看不到。

- **Application** = 商场大厅的公告牌。所有顾客都看到同一块牌子上写着“今日营业时间 10:00-22:00”，谁走谁留公告牌都不变，直到商场自己换牌子。

实际开发中的典型用法：

```python
# 伪代码示例

# Cookie：存用户的界面偏好（客户端的，不敏感）
response.set_cookie("theme", "dark", max_age=30*24*3600)

# Session：存当前用户是否登录（服务端的，敏感）
session["user_id"] = 12345
session["role"] = "admin"

# Application：统计所有用户的访问次数（全局共享）
application["visit_count"] = application.get("visit_count", 0) + 1
```

💡 **现代趋势**：

- (1)Cookie 仍然在用，但敏感数据不会直接存 Cookie，一般是存一个加密的 Session ID 或 JWT Token。


- (2)Session 在传统后端（如 ASP、PHP）中很常用，但在微服务 / 前后端分离架构中，**JWT Token 逐渐取代了 Session**（因为 Session 需要服务器存状态，分布式环境下不好共享）。


- (3)Application 在现代开发中对应的概念叫**应用级缓存**或**全局变量**，在 Python/Java/Node 里都有对应实现，但名字可能不叫 Application（比如 Django 里可以用 `cache.set()` 实现类似效果）。

## 7. 数据库——数据到底存哪了

### 7.1 数据库如何存数据

数据库就是“专门存数据的超级 Excel”，比 Excel 强在：

- 能存几亿条
- 毫秒级查询
- 支持成千上万人同时访问
- 程序关了数据不丢

>数据库（指关系型数据库）把数据存成**二维表**的样子，就像Excel里一张Sheet——**行（Row）**代表一条记录，**列（Column）**代表一个字段（比如用户名、密码）。

>但这些表并不是一个一个独立的文件，而是一个**整体**。你有一个数据库文件（或者一组文件），里面包含了所有表、表之间的关系、索引等。比如你写论文时，你的Word文档是一个文件，里面有章节、段落、图片——数据库同理，它就是硬盘上一个（或一组）文件，数据库软件（MySQL/PostgreSQL/SQL Server）负责管理这个文件里的数据。

>为什么它比Excel快？因为它做了两件事：**建立索引**（像书的目录，不用翻完整本书就能找到内容）和 **预定义结构**（每列的数据类型是固定的，比如“这一列是整数、那一列是字符串”，所以查询时不需要猜测数据格式，直接按固定格式扫描）。

>所以你新建一个数据库时，本质上是在硬盘上创建了一个文件夹或一组文件，里面存放着所有的表结构和数据——它就是你的“数据仓库实体”。

---

### 7.3 SQL——和数据库对话的语言

你只需眼熟四个操作：
- **查**：`SELECT * FROM papers WHERE title LIKE '%农业%';`
- **增**：`INSERT INTO users (username, password_hash) VALUES ('lisi', 'xxx');`
- **改**：`UPDATE papers SET title = '新标题' WHERE id = 1;`
- **删**：`DELETE FROM papers WHERE id = 3;`

但在真实项目里，**你几乎不用手写 SQL**，因为有 ORM。

---

### 7.4 ORM——用代码代替 SQL

**与原生SQL对比：**

|    对比项    |             **原生SQL**             |              **ORM**              |
| :-------: | :-------------------------------: | :-------------------------------: |
|  **建表**   |    手写 `CREATE TABLE` 在SSMS里执行     |    写Python类（Model），ORM自动生成建表语句    |
| **查询数据**  |    手写 `"SELECT * FROM..."` 字符串    | 写 `session.query(类名).filter(...)` |
|  **取数据**  | `row[1]` 或 `row["title"]`（下标或字段名） |     `paper.title`（对象的属性，智能提示）     |
|  **安全性**  |      拼字符串容易出SQL注入漏洞（如果没参数化）       |        ORM自动参数化，基本杜绝SQL注入         |
| **数据库切换** |    SQL Server的语法换到MySQL要改SQL语句    |         ORM代码几乎不用改，只改连接配置         |
| **适合场景**  |       复杂查询、性能要求极高、需要精细控制SQL       |        常规CRUD、快速开发、不想手写SQL        |

用一句话概括：你操作的是SQL字符串，数据库返回的是“行（row）”给你；ORM操作的是Python对象，数据库返回的东西被ORM自动包装成了对象。

ORM 让你用“面向对象”的方式操作数据库，更安全、更直观。比如用 Python 的 SQLAlchemy：

```python
papers = Paper.query.filter(Paper.title.contains("农业")).all()
```

**为什么现在推荐用ORM？**

1. **不用在两个工具之间来回切**：你之前建表在SSMS里写，查询在app.py里写。用ORM后，建表、查询、增删改全在Python代码里完成，不用打开SSMS。
    
2. **代码可读性高**：`session.query(Paper).filter(Paper.author == "张三")` 比 `"SELECT * FROM papers WHERE author = '张三'"` 更接近人类思维。
    
3. **切换数据库代价极低**：你的SQL Server项目想换成MySQL，用原生SQL的话你得把 `SELECT TOP 10` 改成 `SELECT ... LIMIT 10`，一堆语法要改。用ORM的话只改配置文件里的连接字符串，代码一行不动。

## 8. HTTP——前后端怎么“沟通”

前端跑在浏览器，后端跑在服务器，它俩靠 **HTTP 协议**沟通。你可以把 HTTP 理解成“快递员”——前端把请求打包交给快递员，快递员送到后端，后端处理完把结果打包让快递员带回去给前端。快递员不管包裹里装的是什么，只负责按规则送到。

那快递员怎么知道“你这次是要干啥”呢？靠的就是请求里的 **方法** 和 **状态码**。

---

### 8.1 请求方法——你要做什么

每次请求，前端都得告诉后端“我这次是来干嘛的”。HTTP 定义了几种标准动作，最常用的是这四个：

|     方法     |  含义  |          生活类比           |               例子               |
| :--------: | :--: | :---------------------: | :----------------------------: |
|  **GET**   | 拿数据  |    你去图书馆查某本书在不在，只查不借    | `GET /papers?keyword=农业`（查询论文） |
|  **POST**  | 提交数据 |   你填了借书单，把单子交给管理员录入系统   |    `POST /login`（提交账号密码登录）     |
|  **PUT**   | 更新数据 | 你发现之前填的借书信息有误，拿回来改掉再交回去 |          修改论文标题（全量更新）          |
| **DELETE** | 删除数据 |    你把借的书还回去，系统删掉借阅记录    |             删除一篇论文             |
**注**：GET 只是“看”，POST 是“给”，PUT 是“改”，DELETE 是“删”。GET 请求的参数会直接出现在网址里（`?keyword=农业`），所以不适合传密码；POST 的参数放在请求体里，不会在地址栏露出来，相对安全。

---

### 8.2  状态码——服务器给你的“回答暗号”

后端处理完请求后，会返回一个三位数字，告诉前端“结果咋样”。这就像快递员送货后你签收时写个反馈码：200 表示“收到了，没问题”，404 表示“东西找不到了”，500 表示“快递员路上摔了”。

|   状态码   |          含义           |       通俗解释       |                            遇到的典型场景                            |
| :-----: | :-------------------: | :--------------: | :-----------------------------------------------------------: |
| **200** |          OK           | 一切正常，你要的数据在响应体里  |                前端请求 `/papers`，后端正常返回了论文列表 JSON                |
| **401** |    Unauthenticated    | 你没有登录，或者登录凭证已过期  |                    直接访问 `/admin` 页面，但你没登录                     |
| **403** |       Forbidden       | 你已经登录了，但身份不够格干这事 |                   普通用户试图删除论文，系统拒绝：“你不是管理员”                    |
| **404** |       Not Found       |    你要找的资源不存在     |            访问 `/papers/9999`，但数据库里没有 id 为 9999 的论文            |
| **500** | Internal Server Error |   后端代码出 bug 了    | 你在 `app.py` 里写了个空指针，数据库连接失败了，或者 SQL 语法写错了——这是**你调试时最常遇到的状态码** |

**这五个状态码覆盖了 Web 开发中 95% 的场景。** 你在浏览器按 F12 → Network 面板，看到红色状态码时，先对照这张表判断是哪一类问题，再去查对应环节的代码，能省下大量排查时间。

### 8.3 JSON——数据传输的方言

#### 8.3.1 JSON的本质

前端和后端“说话”，总得用一种双方都能听懂的语言。早些年大家用 XML（就是那种一堆尖括号包着的格式），但它太啰嗦了——传输同样的数据，XML 的字符量可能比 JSON 多一倍。

JSON是纯文本，是一个文本文件，用记事本打开能看到字母、冒号、花括号、引号，没有任何神秘的东西。**JSON 不是代码**，没有逻辑判断、没有循环、没有函数。它只是一串有格式的字符，专门用来“描述数据长什么样”。

现在前后端传数据几乎都用 JSON：

```json
{
  "name": "南农图书馆",
  "location": "南京",
  "departments": ["信息管理", "农学"]
}
```

---

#### 8.3.2 JSON的三种引入形态

**（1）形态一：独立的 `.json` 文件**

你的项目文件夹里可能会有一个 `config.json` 或 `data.json` 文件，里面全是 JSON 格式的内容。比如一个前端项目的 `package.json` 就是 JSON 文件。

**（2）形态二：写在代码文件里的字符串**

在 `app.py` 里，你可能会写：
```python
user_data = '{"name": "张三", "age": 20}'
```

这个字符串的内容是JSON格式，但外面包了一层引号——对Python来说，它就是个字符串变量。类似地，JS文件里你可以写 `const data = {"name": "张三"}`（JavaScript里花括号直接是对象，不是字符串，后面会细讲）。

**（3）形态三：网络传输时的“临时文本”**

浏览器向后端发请求，请求体里有一串文本 `{"username":"zhangsan"}`——这串文本是JSON格式。后端收到后，把它解析成Python字典；后端返回响应时，又把Python字典转成JSON格式的文本字符串返回给前端。前端收到后再解析成JS对象。**这个时候JSON只存在于网络传输的那一刹那和两端的缓存区里，没有落地成文件。**

---

#### 8.3.3 JSON的优点

1. **轻量**：没有多余的闭合标签，传输快。
    
2. **易读**：键值对的形式，人看一眼就知道在表达什么。
    
3. **通用**：几乎所有编程语言都有解析 JSON 的库，Python 用 `json` 模块，JavaScript 用 `JSON.parse()`，前后端都能直接处理。

---

#### 8.3.4 在 Web 系统中，JSON具体应用场景？

- **后端返回数据时**：API 接口的响应体（Response Body）里放的就是 JSON，比如 GET `/papers` 返回论文列表。
    
- **前端发送数据时**：POST 请求的请求体（Request Body）里也放 JSON，比如登录接口传 `{"username": "zhangsan", "password": "123456"}`。

你打开浏览器 F12 → Network 面板，随便点一个请求，看看“Response”标签页里显示的内容——十有八九就是一段 JSON。它就是你日常调试 Web 应用时最常打交道的数据格式。

---

### 8.4 **XML：被淘汰的可扩展标记语言**

**XML 的本质定义：**

XML 是可扩展的、自定义标签的标记语言，核心作用是「存数据、传数据」，不是用来做页面显示的。

**和 HTML 的本质区别：**

| 对比项  |                HTML                 |                       XML                       |
| :--: | :---------------------------------: | :---------------------------------------------: |
|  标签  | 固定的、预定义的，只能使用 `<p>`、`<table>` 等官方标签 | 完全自定义，可以自己定义标签，例如 `<book>`、`<student>`、`<身份证号>` |
| 核心作用 |            显示网页内容，给用户查看             |                存储、传输数据，供程序读取和处理                 |
| 大小写  |       不敏感，`<P>` 和 `<p>` 效果相同        |        极度敏感，`<name>` 和 `<Name>` 是两个不同标签         |
| 语法要求 |        相对宽松，部分错误情况下浏览器仍可能显示         |             非常严格，标签错误或格式不规范会导致解析失败              |
HTML 是「给人看的页面语言」，XML 是「给程序读的数据语言」。比如你要存一本书的信息：

-  HTML 写出来是给人看的：
```html
<p>书名：西游记 作者：吴承恩</p>
```

- XML 写出来是给程序读的：
```html
<book><name>西游记</name><author>吴承恩</author></book>
```
程序可以直接提取书名、作者的值。

**现在JSON比XML更常用，但配置文件（如`web.xml`）、Office文档（`.docx`本质是XML压缩包）还在用XML，看到认识就行。**

---

### 8.5 Ajax与异步请求

#### **8.5.1 Ajax 的本质定义**

Ajax 不是一门新技术，是几种老技术组合起来的新用法，全称是「异步 JavaScript 和 XML」。 它的核心价值只有一个：不用刷新整个页面，就能更新页面的一部分内容。

---

#### **8.5.2  传统 Web vs Ajax 的本质区别**

|      **传统 Web 模式**       |         **Ajax 模式**          |
| :----------------------: | :--------------------------: |
| 你点按钮→整个页面全部刷新，白屏一下再显示新内容 | 你点按钮→只刷新你要改的那一小块，其他地方不动，没有白屏 |
| 同步：你必须等服务器返回结果，期间什么都干不了  |   异步：你等结果的时候可以继续点别的、干别的，不卡   |
|     每次都要加载整个页面，费流量、慢     |       只传要改的那点数据，省流量、快        |
举例：
- 传统模式：你刷朋友圈，点「加载更多」，整个页面刷新，回到顶部，你要重新往下滑
- Ajax 模式：你刷朋友圈，点「加载更多」，下面直接多出新的朋友圈，上面你看过的内容完全不动，也不会回到顶部

现在用的所有现代网站（微信、淘宝、抖音、百度），所有不用刷新页面的交互，全是 Ajax 做的。

---

####  **8.5.3 Ajax 的核心技术组成**

- JS：负责触发事件、处理数据
-  XMLHttpRequest：核心对象，负责和服务器偷偷传数据，不用刷新页面。现在实际开发中，`XMLHttpRequest` 已被更简洁的 `fetch()` 替代，但原理完全相同。你看到 `fetch()` 的时候知道它就是在做 Ajax 的事就行。
- DOM：负责拿到数据后，只改页面的对应部分
- XML/JSON：用来传数据（现在基本都用 JSON，不用 XML 了）

## 9. 用户登录与权限——网站必备功能

前几章我们拆解了 Web 系统的所有零件：前端负责界面、后端负责逻辑、数据库负责存储、HTTP负责通信。之后我们不学新零件，现在的目标是**把这些零件组装起来，做一个所有网站都有的功能——用户登录**。

读完这一章，你会看到前端、后端、数据库、HTTP 是怎么连成一条线的。它不是新概念，它是概念落地的结果

---

### 9.1 登录流程——从输入密码到进入系统

一个完整的登录流程，按时间顺序拆开看是这样的：

```
用户输入账号密码 → 点击登录
    ↓
前端把数据打包成 JSON，发 POST 请求给后端
    ↓
后端接收请求，从请求体里拿出账号和密码
    ↓
后端去数据库的 users 表里查这个账号存不存在
    ↓
如果账号存在，比对密码对不对（注意：比的是哈希值，不是明文）
    ↓
如果密码正确，后端生成一个 Token（身份凭证）返回给前端
    ↓
前端收到 Token，把它存起来（存在 localStorage 或 Cookie 里）
    ↓
之后用户再访问其他页面（比如论文列表），前端每次都在请求里带上这个 Token
    ↓
后端每次收到请求，先验证 Token 是否有效、是否过期
    ↓
验证通过 → 后端知道“这是谁在操作”，返回对应的数据
```

**每一步为什么要这么做？**

**（1）为什么密码不能明文存数据库？**

用户注册时填的密码，如果直接存进数据库，那谁拿到数据库就能看到所有人的密码。正确的做法是存 **密码的哈希值**——把明文密码通过一个单向函数（比如 bcrypt）变成一串固定长度的乱码。这个函数的特点是：同样的明文每次算出来的哈希都一样，但知道哈希值无法反推出明文。

所以登录时比对的不是“用户输入的密码”和“数据库存的密码”，而是“把用户输入的密码做哈希，看哈希值是否匹配”。

**（2）为什么登录成功后不直接把“用户已登录”状态记在后端，而是要返回一个 Token 给前端？**

因为 HTTP 是无状态的——服务器记不住每个请求是谁发的。你登录成功后，后端说“好了你登录了”，但下次你发请求时，后端已经忘了你是谁。所以需要给前端一个“凭证”，前端每次请求都带着它，后端才能认出你。

**（3）Token 存在前端哪里？**

通常存在浏览器的 `localStorage`（本地存储，关掉浏览器再打开还在）或 `Cookie`（会自动随请求发送）里。存 `Cookie` 的好处是浏览器自动带过去，存 `localStorage` 的好处是更灵活、不受 Cookie 大小限制。

---
### 9.2 Token 和 JWT

#### 9.2.1 Token 是什么？

Token 就是一串加密后的字符串，相当于“电子身份证”。你登录成功后，后端给你发一张“身份证”，之后你去哪都带着它，后端一验就知道是你。

---

#### 9.2.2 JWT 是什么？

JWT（JSON Web Token）是目前最流行的 Token 格式。它长这样：

```text
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJ1c2VyX2lkIjoxLCJyb2xlIjoidXNlciJ9.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c
```

看起来是一串乱码，实际上它由三部分组成，用点号 `.` 隔开：

| 部分  |      名称       |                   存的内容                   |        能不能看         |
| :-: | :-----------: | :--------------------------------------: | :-----------------: |
| 第一段 |  Header（头部）   |            加密算法信息，比如 `HS256`             | 能看（Base64解码后是 JSON） |
| 第二段 |  Payload（载荷）  | 用户信息，比如 `{"user_id": 1, "role": "user"}` |   能看（但不要存密码等敏感信息）   |
| 第三段 | Signature（签名） |            前两段 + 一个密钥 算出来的签名             |        不能伪造         |
因为 JWT 里**本身就带着用户信息**（user_id、role、过期时间等）。后端拿到 JWT，不用再去数据库查一遍用户信息，直接解码 JWT 就知道是谁。

这就和 Session 方式形成了对比：Session 方式下，后端需要存一份“Session ID → 用户信息”的映射表，每次请求都要查表；JWT 方式下，后端什么都不用存，Token 里自带一切。

简单来说：
- Session = 我去南农外洗浴中心，前台给我一个手牌，手牌上只有一个号码，我每次来前台都要去柜子里翻我的档案。（后端存状态，每次都要查）
    
- JWT = 我去南农外洗浴中心，前台给我一张卡，卡上写着“姓名、SVIP会员、有效期2029年底”，保安扫一眼就知道让我进。（后端不存状态，Token 自带信息）

---

#### 9.2.3** JWT 的优缺点**

|            优点            |                  缺点                  |
| :----------------------: | :----------------------------------: |
|     服务器不用存状态，适合分布式系统     |     Token 一旦签发，到期前无法撤销（除非加黑名单机制）     |
| 跨服务认证方便（多个服务共享同一个密钥就能验证） | Token 体积比 Session ID 大，每次请求都带着有点浪费带宽 |
|       扩展性好，适合微服务架构       |          如果密钥泄露，谁都能伪造 Token          |

---

#### 9.2.4 JWT vs Session对比表

| 对比项   | Session       | JWT          |
| ----- | ------------- | ------------ |
| 状态存储  | 服务器（内存/Redis） | 客户端（Token自身） |
| 分布式支持 | 差，需共享存储       | 好，任何服务都能验证   |
| 典型场景  | 单体应用、传统后端渲染   | 前后端分离、微服务    |

---
### 9.3 权限控制

登录只是确认“你是谁”，权限控制决定“你能干什么”：
- **普通用户**：只能看、搜
- **管理员**：能删、能改

后端代码里会这样判断：

```python
if current_user.role == "admin":
    # 允许删除
else:
    # 返回 403 禁止访问
```

| 前端图标  | FontAwesome（CDN）  |                  图标库，加一个 `<i>` 标签就能显示图标                  |        浏览器加载         |
| 前端交互  | 原生 JavaScript（少量） |              只有收藏、引用搜索等少数功能需要异步请求，不需要引入整个框架              |     直接写在 HTML 里      |
| 后端框架  |  Python + Flask   |                一开始学的就是这个，课设用它跑通过的，不想重新学一套                |       PyCharm        |
|  数据库  | SQL Server + SSMS |                  南农数据库课教的就是这个，用着顺手，不折腾                   |         SSMS         |
| 数据库访问 |  原生 SQL（pyodbc）   | 我会写 SQL，用原生 SQL 我能精确控制查什么、怎么查。ORM 多一层翻译，出错了我还得去查 ORM 的文档 |        pyodbc        |
| 版本控制  |   Git + GitHub    |                      记录历史版本，方便备份和展示                      | VS Code Git 面板 / 命令行 |


**选型原则**：用什么不是比赛，你做得顺手的组合就是最好的组合。


**为什么不用 Vue/React？**

Vue/React 适合需要复杂交互的页面（比如在线文档编辑器、实时聊天工具、数据大屏）。我的系统是一个文献检索系统，核心操作是：搜索 → 看结果 → 点进去看详情 → 下载。这种“展示型”页面用服务端渲染就够了，用 Vue 相当于“杀鸡用牛刀”——增加了项目复杂度、需要额外学一套语法、还要配置前后端联调环境。

**为什么不用 ORM（SQLAlchemy）？**

我手写 SQL 已经能熟练做到的事情，ORM 只是换了一种写法，没有带来额外的收益，反而增加了需要学习的内容和定位问题的时间。如果我的项目需要频繁切换数据库（比如从 SQL Server 切到 PostgreSQL），ORM 才有明显的价值。我目前没有这个需求。

**为什么不用前后端分离？**

前后端分离适合的项目：多人协作（前端一组人、后端一组人）、大团队分工、需要同时维护多个前端（Web + App + 小程序）。

我的情况是：一个人开发一个毕设系统。后端渲染模式（Flask 直接把 HTML 返回给浏览器）意味着我少配一套跨域、少跑一个前端服务、少学一套 Vue 语法。整个请求链路是：浏览器 → Flask → SQL → Flask → 浏览器。我能看到完整的路径，出了 bug 也知道去哪查。

#### 10.3.6 后端怎么连数据库

在 `app.py` 顶部写一个连接函数：

```python
import pyodbc

def get_connection():
    return pyodbc.connect(
        'DRIVER={SQL Server};'
        'SERVER=destiny;'          # 你的 SQL Server 实例名
        'DATABASE=AcademicSearchDB;'
        'Trusted_Connection=yes;'
    )
```

然后在每个路由里调用它：

```python
@app.route('/papers')
def papers():
    conn = get_connection()
    cursor = conn.cursor()
    cursor.execute("SELECT * FROM Documents")
    results = cursor.fetchall()
    conn.close()
    return render_template('papers.html', papers=results)
```

## 12. 报错了怎么办——排查思路速查

这一章按“报错长啥样 → 啥意思 → 怎么办”来写，遇到问题直接翻。

**调试前的第一件事：看清楚报错信息在说什么。**
- 红色字不等于“你完了”，它是“这里出问题了”，认真读，很多时候答案就在最后几行。
- 前端报错看浏览器 F12 的 **Console** 面板。
- 后端报错看运行后端的终端窗口（PyCharm 下面的 Run 面板）。


### 12.1 前端相关报错
| 报错关键词                                     | 什么意思                      | 怎么解决                                     |
| ----------------------------------------- | ------------------------- | ---------------------------------------- |
| `Module not found`                        | 缺依赖包                      | `npm install [包名]`，报错里会告诉你是哪个包           |
| `Failed to resolve import`                | 文件路径写错了                   | 检查 `import` 语句里的路径是否正确（`./` 和 `../` 别搞混） |
| `Port 5173 is already in use`             | 端口被占用（另一个项目占用了）           | 换端口：`npm run dev -- --port 5174`         |
| `Cannot read property 'xxx' of undefined` | 变量是 `undefined` 但你把它当对象用了 | 用 `?.` 可选链，或加 `if (data && data.xxx)` 判断 |
| `xxx is not defined`                      | 用了没定义的变量                  | 检查变量名是不是拼错了，或者忘了声明                       |

### 12.2 后端相关报错
| 报错关键词 | 什么意思 | 怎么解决 |
|-----------|----------|----------|
| `No module named 'xxx'` | Python 缺包 | `pip install xxx`，然后把包名加到 `requirements.txt` 里 |
| `Connection refused`（数据库） | 数据库服务没开，或者连接地址/端口写错了 | 检查 SQL Server 服务是否在运行，检查 `SERVER=xxx` 里的服务器名对不对 |
| `KeyError: 'xxx'` | 字典里没有这个键 | `print(字典对象)` 看看到底有哪些字段，再对照代码改 |
| `TypeError: 'NoneType'` | 某个变量是 `None`，但你把它当列表/对象用了 | 检查查询结果是否为空，加 `if result is None` 判断 |
| `ImportError` / `ModuleNotFoundError` | 包没装上，或者路径不对 | 检查虚拟环境是否激活，检查 `sys.path` |

### 12.3 联调相关报错
| 报错关键词                                   | 什么意思                | 怎么解决                                                                                 |
| --------------------------------------- | ------------------- | ------------------------------------------------------------------------------------ |
| `CORS policy`                           | 跨域被浏览器拦截            | 后端装 `flask-cors`（Flask）或 `fastapi.middleware.cors`（FastAPI）并配置                       |
| `404 Not Found`                         | URL 写错了，或者后端路由没写对   | 检查请求 URL 是不是 `http://localhost:5000/papers`，检查后端有没有 `@app.route('/papers')`          |
| `500 Internal Server Error`             | 后端代码报错了（最常见）        | **看后端终端的红色堆栈**，从下往上读，找到第一个 `File "/你的项目/app.py"` 那一行                                 |
| `ERR_CONNECTION_REFUSED`                | 后端服务根本没启动           | 去 PyCharm 点运行，或者终端执行 `python app.py` / `uvicorn main:app --reload`                   |
| `401 Unauthenticated` / `403 Forbidden` | 认证/权限问题（没登录或权限不够）   | 参考第8.2节状态码说明，检查登录状态和用户角色                                                             |
| 后端返回200但页面空白                            | 后端查不到数据，或者前端渲染条件没触发 | 先看Network里Response到底返回了什么。如果是`[]`或`null`，去后端查数据库；如果数据有但页面空白，检查前端`v-for`或`if`条件是不是写错了 |


**跨域与404同时出现时怎么办？**
先解决跨域（CORS），再解决404。因为跨域被拦截时浏览器根本不发请求到后端，你看不到后端到底有没有收到请求。跨域通了之后，如果还是404，再看URL对不对。

### 12.4 Flask 专用报错排查
| 报错关键词 | 什么意思 | 怎么解决 |
|-----------|----------|----------|
| `werkzeug.routing.BuildError` | `url_for` 里写的路由名字不存在 | 检查 `@app.route` 里的函数名和 `url_for` 里写的是不是一致 |
| `jinja2.exceptions.TemplateNotFound` | 模板文件没找到 | 检查 `templates/` 文件夹里有没有对应的 `.html` 文件 |
| `sqlalchemy.exc.OperationalError` | 数据库连接失败 | 检查 `SERVER` 和 `DATABASE` 配置是否正确，SQL Server 服务是否启动 |
| `pyodbc.Error` 或 `ProgrammingError` | SQL 语法写错了，或者表名/字段名写错了 | 把 SQL 语句打印出来（`print(sql)`），复制到 SSMS 里跑一遍，看报什么错 |

### 12.5 万能排查流程
遇到任何报错，按这个顺序查：

1. **看报错信息**：红色字就是答案，认真读，不要跳过。最后几行往往直接告诉你“哪里错了”。
2. **看终端/控制台**：
   - 前端报错看浏览器 F12 → Console 面板。
   - 后端报错看 PyCharm 下面的 Run 面板（终端窗口）。
3. **二分法排除**：
   - 用 Postman 或浏览器直接访问后端接口（比如 `http://localhost:5000/papers`）。
   - 如果 Postman 能通但前端不行 → 问题在前端（跨域或请求写法）。
   - 如果 Postman 都不通 → 问题在后端（路由或代码逻辑）。
4. **打印大法（Print Debugging）**：
   - 在关键位置加 `print()`，看变量值到底是什么。
   - 在 Flask 里打印 SQL 语句：`print(sql)`，复制到 SSMS 里跑一遍。
5. **复制报错给 AI**：
   - 直接粘贴整段报错，AI 能帮你定位 80% 的问题。
   - 用第11.5节的万能报错模板，把代码和报错一起贴过去。

> 💡 **万能提示词**：“我遇到一个 bug，现象是 `[描述现象，比如'点登录按钮后页面没反应']`，报错是 `[粘贴完整的红色报错堆栈]`，代码是 `[粘贴关键代码]`，请帮我定位并修复。”
